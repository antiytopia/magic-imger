# Architecture Notes

## Цель

Сделать один общий движок обработки изображений и не дублировать бизнес-логику между платформами.

## Базовый подход

Архитектура должна быть слоистой, но очень компактной:

1. `core` отвечает за обработку изображений;
2. `cli` отвечает за аргументы и запуск задач;
3. `windows-ui` в будущем вызывает тот же `core` или CLI-команду.

## Почему так

Если писать отдельно GUI-логику и отдельно CLI-логику с собственной обработкой файлов, проект быстро потеряет главное преимущество: минимальный код.

Поэтому правильный путь для MVP:

- один движок;
- один набор правил обработки;
- один формат входных параметров;
- разные интерфейсы поверх одного и того же поведения.

## Предлагаемая структура проекта

```text
src/
  cli/
    index.ts
    commands/
      resize.ts
      compress.ts
      convert.ts
  core/
    pipeline.ts
    formats.ts
    fs.ts
    jobs.ts
  shared/
    types.ts
```

## Ответственность модулей

### `core/pipeline.ts`

Собирает пайплайн `sharp` для:

- resize;
- compress;
- convert.

### `core/formats.ts`

Содержит правила по форматам:

- какие форматы поддерживаются;
- где есть `quality`;
- где возможен `lossless`;
- какие параметры доступны для `jpg/png/webp/avif`.

### `core/fs.ts`

Отвечает за файловую дисциплину:

- чтение входной папки;
- фильтрация изображений;
- создание выходной директории;
- сохранение имен файлов без хаоса.

### `core/jobs.ts`

Запускает batch-обработку:

- проход по файлам;
- ограничение параллелизма при необходимости;
- единый вывод статусов и ошибок.

## Кроссплатформенный план

### Linux и macOS

Один CLI, одинаковый сценарий использования.

Это самый надежный вариант для MVP, потому что:

- меньше кода;
- меньше расхождений поведения;
- проще отлаживать;
- легче документировать.

### Windows

Если нужен UI, лучше делать его как тонкий слой над тем же ядром.

Подход:

1. пользователь заполняет форму;
2. UI валидирует ввод;
3. UI передает параметры в общий обработчик;
4. обработчик выполняет ту же логику, что и CLI.

## Решение для macOS

Если позже все же понадобится удобный интерфейс, разумный вариант:

- оставить CLI как основной и поддерживаемый путь;
- при необходимости добавить очень тонкий desktop wrapper только после появления возможности тестирования на macOS.

До этого момента отдельный macOS GUI не рекомендуется.

## Принципы API команд

Команды должны быть читаемыми и предсказуемыми:

```bash
magic-imger resize <input> --width 1200 --height 1200 --fit contain --output ./dist
magic-imger resize <input> --width 1200 --height 1200 --fit cover --crop center --output ./dist
magic-imger compress <input> --quality 82 --output ./dist
magic-imger convert <input> --to avif --quality 55 --output ./dist
```

## Минимум решений для MVP

- поддержать только `jpg/png/webp/avif`;
- поддержать только нужные параметры;
- не раздувать CLI десятками флагов;
- не добавлять GUI на все платформы сразу;
- не смешивать файловую логику и обработку изображения в одном месте.

## Критерий качества

Другой разработчик должен открыть проект и за несколько минут понять:

- где лежит обработка;
- где лежит CLI;
- как добавляется новый формат;
- как работает batch mode;
- что именно считается MVP, а что нет.
