# Architecture Notes

Принцип

 один общий движок обработки изображений и несколько интерфейсов поверх него, не дублируя бизнес-логику.
После обновления ТЗ это означает:

- Linux и macOS используют CLI;
- Windows получает полноценный GUI;
- и CLI, и GUI работают через один и тот же `core`.

## Главный архитектурный принцип

Красивое Windows-окно не должно означать вторую реализацию пайплайна.

Правильная схема такая:

1. `core` знает, как обрабатывать изображения;
2. `cli` знает, как принимать аргументы из терминала;
3. `windows-ui` знает, как собирать визуальный ввод пользователя;
4. оба интерфейса передают параметры в одну и ту же доменную модель задач.

## Роли слоев

### `core`

Отвечает за все, что связано с обработкой:

- resize;
- compress;
- convert;
- batch mode;
- файловые правила;
- форматы;
- применение общих и индивидуальных настроек;
- ресурсные ограничения и safe mode;
- предварительную оценку времени и ресурсоемкости.

Также `core` содержит подсистему скриншотов (`core/screenshots`), которую используют и CLI, и GUI.

### `cli`

Отвечает только за:

- разбор аргументов;
- валидацию ввода на уровне CLI;
- запуск задач через `core`;
- понятный вывод результатов и ошибок.

### `windows-ui`

Отвечает только за:

- визуальную очередь файлов;
- drag and drop;
- импорт из буфера обмена;
- редактирование общих параметров;
- редактирование индивидуальных параметров;
- отображение прогресса, ошибок и результата;
- показ preflight estimate перед стартом.

## Скриншоты (URL screenshots)

Логика сделана отдельной подсистемой `core/screenshots`:

- `run-shot-batch.ts` — батч по списку URL и общая структура результата;
- `make-shot.ts` — захват кадров одной страницы (скролл/снимок/оптимизация PNG);
- `browser.ts` — выбор и запуск браузера (bundled/system/custom/CDP).

## Доменные сущности

Чтобы GUI и CLI говорили на одном языке, нужны общие структуры данных.

### `InputAsset`

Описывает входной элемент:

- путь к файлу;
- тип источника: файл, папка, буфер обмена;
- имя;
- исходный формат;
- исходные размеры;
- служебный идентификатор в очереди.

### `JobPreset`

Описывает общие настройки для пачки:

- действие или набор действий;
- target format;
- width;
- height;
- fit;
- crop;
- quality;
- lossless;
- output folder;
- overwrite.

### `JobOverride`

Описывает индивидуальные изменения для одного элемента очереди.

Смысл:

- по умолчанию файл наследует общий preset;
- при необходимости пользователь переопределяет часть параметров;
- итоговая конфигурация собирается как `preset + override`.

### `ResolvedJob`

Финальная задача, которую исполняет `core`.

Она уже не зависит от того, пришла она из CLI или из GUI.

### `ResourceProfile`

Описывает режим исполнения:

- `safe`;
- `balanced`;
- будущие режимы при необходимости.

Для MVP главным считается `safe`.

Для MVP профили должны быть жестко определены:

- `safe`: до `1 GB RAM`, `1` processing job, `1` preview job;
- `balanced`: до `1.5 GB RAM`, `2` processing jobs, `1` preview job.

### `PreflightEstimate`

Описывает предварительный расчет перед запуском:

- число файлов;
- типы исходных форматов;
- целевой формат;
- расчетный профиль ресурсов;
- примерный диапазон времени выполнения;
- ожидаемый диапазон RAM;
- ожидаемый speed gain относительно `safe`, если доступен `balanced`;
- предупреждения о тяжелых местах.

## Обязательная логика Windows UI

Windows UI теперь является частью MVP, а не второстепенной надстройкой.

### Требования к очереди

Очередь должна поддерживать:

- добавление файлов;
- добавление папки;
- drag and drop;
- вставку из буфера обмена;
- удаление элемента;
- очистку очереди;
- множественный выбор;
- жесткий лимит: не более `100` файлов в одной очереди.

### Требования к представлению

Для каждого элемента очереди нужно показывать:

- превью;
- имя;
- формат;
- размер файла;
- размеры изображения;
- текущий статус;
- признак индивидуальных настроек.

### Требования к настройке

UI должен уметь:

- применять общие параметры ко всей очереди;
- открывать инспектор отдельного элемента;
- задавать индивидуальные override-параметры;
- сбрасывать override обратно к общим настройкам.

### Требования к исполнению

Во время запуска UI должен:

- собирать `ResolvedJob` для каждого элемента;
- запрашивать `PreflightEstimate` до старта;
- показывать пользователю ориентир по времени и ресурсам;
- передавать задачи в `core`;
- получать прогресс;
- показывать ошибки на уровне конкретного файла;
- после завершения показывать итоговый путь сохранения.

## Пайплайн обработки

`core/pipeline.ts` должен строить цепочку `sharp` на основе финальной задачи.

Порядок для MVP:

1. загрузка входного изображения;
2. resize, если он включен;
3. compress-настройки, если они нужны;
4. convert в целевой формат;
5. запись в output.

## Batch mode

`core/jobs.ts` отвечает за пакетную обработку.

Что он должен делать:

- принимать список `ResolvedJob`;
- отклонять очередь больше `100` файлов;
- запускать обработку с ограничением параллелизма;
- использовать дефолтную конкурентность `2` для CLI;
- не превышать потолок конкурентности `4` даже на мощной машине в MVP;
- отдавать статусы;
- не валить всю очередь из-за одного битого файла;
- возвращать подробный итог по каждому элементу.

## Safe mode как дефолтная стратегия GUI

Если цель: не съедать ресурсы пользователя, Windows GUI должен работать не в максимально быстром, а в максимально предсказуемом режиме.

### Политика safe mode

- `safe` это дефолтный профиль Windows UI;
- в `safe` активен только `1` processing job;
- preview jobs идут отдельно и тоже с лимитом `1`;
- UI не строит полные превью сразу для всей очереди;
- тяжелые операции не исполняются в renderer;
- целевой верхний предел потребления памяти в `safe`: до `1 GB RAM`.

### Что делает `core/resources.ts`

Модуль должен отвечать за:

- выбор конкурентности;
- правила для AVIF;
- приостановку низкоприоритетных задач;
- проверку лимитов очереди;
- оценку риска по RAM/CPU/диску на уровне профиля исполнения;
- запрет запуска safe mode задач, которые прогнозно выходят за `1 GB RAM`.

### Что делает `core/estimator.ts`

Модуль должен отвечать за preflight estimate:

- читать metadata файлов до запуска полной обработки;
- учитывать количество файлов, размеры, форматы и выбранный output;
- давать диапазон времени, а не ложную точную цифру;
- оценивать ожидаемый расход RAM;
- сравнивать `safe` и `balanced`, если пользователь может включить повышенный профиль;
- считать ожидаемый speed gain относительно `safe`;
- маркировать тяжелые сценарии вроде крупных PNG и AVIF.

### Правила backpressure

Система должна:

- не запускать новые processing jobs, пока не освободился слот;
- снижать активность preview jobs во время обработки;
- приоритетно держать отзывчивым UI, а не максимизировать throughput;
- не держать больше одного тяжелого изображения в активной декодированной стадии в safe mode.

### Правила по памяти

- превью должны быть маленькими и ленивыми;
- после завершения файла связанные buffers должны освобождаться как можно раньше;
- clipboard assets должны жить во временной папке, а не в долгоживущем in-memory кеше;
- нельзя держать в памяти весь batch ради скорости.

### Правила по CPU

- AVIF в safe mode всегда обрабатывается последовательно;
- никакого автоматического повышения конкурентности в UI по умолчанию;
- CPU-пиковая нагрузка допускается только от одного активного job, а не от нескольких одновременно.

### Правила по диску

- до старта batch нужно проверять, что есть минимальный свободный запас места;
- временные clipboard файлы должны удаляться после завершения или очистки очереди;
- output folder не должен повторно сканироваться как input.

### Выход за safe limit только явно

Если пользователь хочет дать приложению больше ресурсов:

- в GUI это включается явным переключателем вроде `Allow more resources`, но не безлико, а с конкретной подписью по лимитам;
- в CLI это включается отдельным флагом вроде `--allow-more-resources` или `--resource-profile balanced`.

Без этого явного действия:

- GUI обязан оставаться в профиле `safe`;
- прогнозный расход RAM не должен выходить за `1 GB`;
- запуск тяжелой задачи должен блокироваться с понятным предупреждением.

### Что именно должен показывать GUI-переключатель

Переключатель должен показывать пользователю не только факт "больше ресурсов", но и конкретику:

- текущий профиль;
- RAM budget;
- количество processing jobs;
- ориентир по времени в текущем профиле;
- ориентир по времени в `balanced`;
- ожидаемый speed gain.

Пример формата:

- `Safe: up to 1 GB RAM, 1 active job, ~18 min`
- `Balanced: up to 1.5 GB RAM, 2 active jobs, ~11-13 min`
- `Estimated speed gain: 30-40%`

## Файловая дисциплина

`core/files.ts` отвечает за предсказуемое поведение на диске:

- чтение папки;
- фильтрацию поддерживаемых форматов;
- создание output folder;
- сохранение исходных имен;
- замену расширения при convert;
- защиту от случайной перезаписи без явного флага.

## Поддержка буфера обмена

Буфер обмена нужен только GUI-слою, но результат должен попадать в те же доменные сущности.

Подход:

1. UI получает изображение из clipboard;
2. UI создает временный входной asset или in-memory representation;
3. дальше этот asset обрабатывается так же, как обычный файл;
4. пользователь все равно выбирает output folder и правила сохранения результата.

## Решение для macOS

Так как вы не сможете тестировать отдельный macOS GUI, безопасный путь такой:

- официально поддерживать CLI на macOS в MVP;
- не обещать отдельное нативное окно до появления тестового контура;
- если GUI когда-нибудь понадобится, делать его на том же UI-слое и том же `core`, а не проектировать заново.

## Технологический выбор для Windows UI

Практичный вариант для этого проекта:

- `Electron` как оболочка окна;
- `Node.js` + `sharp` как общий runtime обработки;
- любой легкий UI-слой поверх, если нужен компонентный интерфейс.

Почему это разумно:

- проще работать с файлами, drag and drop и clipboard;
- проще переиспользовать Node-окружение;
- меньше трения, чем в более экзотических схемах;
- легче собрать красивое окно без второй серверной части.

Если позже появится строгая цель уменьшить вес desktop-приложения, это можно пересмотреть, но для MVP `Electron` выглядит прагматично.

## Лимиты и расчет ресурсов

Это ориентир для проектирования, а не обещание точных benchmark-цифр.

### Жесткие лимиты MVP

- максимум `100` файлов в очереди;
- дефолтная конкурентность CLI: `2` активные задачи;
- дефолтная конкурентность Windows GUI в `safe mode`: `1` активная задача;
- safe mode по умолчанию должен удерживаться в пределах `до 1 GB RAM`;
- профиль `balanced` допускается только при явном согласии пользователя и должен удерживаться примерно в пределах `до 1.5 GB RAM`;
- абсолютный потолок конкурентности в MVP: `4`;
- для AVIF-сценариев целевая рабочая конкурентность: `1-2`.

### Допущения для оценки

Расчет ниже исходит из такого сценария:

- `100` файлов в очереди;
- средний размер изображения от `3000x2000` до `4000x3000`;
- типичные входные форматы: `jpg/png/webp`;
- типичный output: `jpg/webp`;
- превью в UI генерируются лениво, а не все сразу в полном размере.

### RAM

Практический ориентир:

- Windows UI с Electron, очередью и превью: `250-450 MB`;
- один активный job на средне-крупном изображении: `100-250 MB` пиково;
- при конкурентности `1` в safe mode: обычно `350-700 MB`;
- safe mode должен проектироваться так, чтобы тяжелые пики оставались в районе `0.9-1.0 GB`, а не выше;
- выход за `1 GB` допустим только в явно разрешенном профиле `balanced`.

Инженерный вывод:

- базовая целевая машина для комфортной работы: `8 GB RAM` и выше;
- на `4 GB RAM` нужен safe mode без отклонений;
- держать низкую конкурентность важнее, чем увеличивать лимит очереди.

### CPU

- `jpg/webp` обычно заметно грузят `1-2` ядра;
- `png` часто грузит и CPU, и память;
- `avif` самый тяжелый по времени кодирования и может временно занимать CPU почти полностью;
- при конкурентности `1` UI будет дольше работать, но поведение системы будет заметно спокойнее;
- при конкурентности `2` высокая CPU-нагрузка считается нормальным поведением.

Инженерный вывод:

- `4` логических потока CPU достаточно для обычного MVP-сценария;
- safe mode нужен именно для защиты слабых машин, а не для скорости;
- `6-8` потоков дают более комфортную работу с AVIF и крупными батчами.

### Оценка ускорения при `balanced`

Прирост скорости нужно считать ориентировочно, а не обещать линейное ускорение.

Практичные диапазоны для preflight estimate:

- `jpg/webp`: обычно около `1.4x-1.8x` быстрее относительно `safe`;
- `png`: обычно около `1.2x-1.5x`;
- `avif`: обычно около `1.1x-1.4x`;
- на слабом CPU или диске выигрыш может быть заметно ниже.

Архитектурное правило:

- estimate должен показывать именно диапазон ускорения или диапазон времени;
- если прогнозный выигрыш меньше примерно `15%`, GUI должен прямо говорить, что повышение лимита почти не даст пользы.

### Диск

- нужно место под исходники;
- нужно место под выход;
- нужен запас под временные clipboard assets и служебные операции.

Практический ориентир:

- пачка из `100` файлов по `8-12 MB` даст входной объем около `0.8-1.2 GB`;
- безопасный запас свободного места для такого запуска: `2-3 GB`.

### Архитектурное следствие

- лимит `100` файлов нужен не потому, что `sharp` не умеет больше, а потому что UI и пользовательская машина не должны страдать от случайного оверлоада;
- реальный контроль нагрузки должен идти через `p-limit`, lazy previews, backpressure и preflight estimate;
- если в будущем понадобится больше `100` файлов, это уже должен быть отдельный режим с другим UX и другой ресурсной моделью.

## Минимум решений для MVP

- поддержать только `jpg/png/webp/avif`;
- оставить одну реализацию `core`;
- не плодить разные пайплайны под GUI и CLI;
- не раздувать настройки до уровня графического редактора;
- не делать отдельный GUI для macOS до появления возможности тестировать.

## Предполагаемые баги и митигации

### EXIF orientation

Риск:

- часть изображений с телефона может открываться повернутой неверно.

Митигация:

- нормализовать ориентацию в `core` до resize и export;
- покрыть тестами портретные и альбомные кейсы.

### Потеря transparency

Риск:

- при конвертации в `jpg` прозрачность исчезает, а пользователь может не ожидать этого.

Митигация:

- предупреждать в UI и CLI;
- заполнять фон предсказуемым цветом или требовать явного подтверждения стратегии.

### Коллизии имен

Риск:

- `a.png` и `a.webp` после конвертации в `jpg` дадут конфликт по имени.

Митигация:

- заранее проверять output map;
- добавлять стратегию именования: overwrite, skip, suffix.

### Повторная обработка output

Риск:

- если output лежит внутри input, batch mode может начать обрабатывать собственный результат.

Митигация:

- исключать output folder из обхода;
- запрещать опасные конфигурации без явного подтверждения.

### Фризы Windows UI

Риск:

- большие изображения или пачки файлов могут подвесить renderer.

Митигация:

- тяжелую обработку держать вне renderer;
- UI должен только отправлять задачи и получать события прогресса.

### Баги clipboard flow

Риск:

- буфер обмена не содержит нормального имени, расширения или стабильного пути;
- некоторые приложения кладут в clipboard не файл, а bitmap-представление.

Митигация:

- генерировать временный asset id;
- использовать контролируемую схему именования при сохранении;
- явно различать clipboard image и clipboard file list.

### Проблемы Windows paths

Риск:

- длинные пути, кириллица, пробелы и locked files могут ломать обработку.

Митигация:

- везде использовать `path` API;
- тестировать Unicode paths;
- отдавать понятные ошибки уровня файла, а не падать всей пачкой.

### Медленный AVIF

Риск:

- пользователю может казаться, что приложение зависло, хотя реально идет тяжелое кодирование.

Митигация:

- показывать прогресс;
- явно маркировать AVIF как более медленный формат;
- ограничивать конкурентность AVIF-задач.

### Неодинаковое поведение quality

Риск:

- одно и то же значение `quality=80` визуально и по размеру ведет себя по-разному в разных форматах.

Митигация:

- хранить правила по форматам в `core/formats.ts`;
- не обещать одинаковую компрессию между форматами;
- использовать разные дефолты для `jpg/webp/avif`.

### Ошибочный прогноз времени

Риск:

- предварительная оценка времени может быть неточной из-за разных типов файлов и производительности машины.

Митигация:

- показывать диапазон времени, а не одну точную цифру;
- явно помечать estimate как ориентир;
- при накоплении статистики в будущем можно улучшать прогноз локально.

### Битые и неподдерживаемые файлы

Риск:

- один поврежденный файл способен сорвать весь batch.

Митигация:

- fail per file, not fail per batch;
- итоговый отчет должен разделять success, skipped и failed.

## Критерий качества

Другой разработчик должен открыть проект и быстро понять:

- где лежит бизнес-логика;
- где лежит CLI;
- где лежит Windows UI;
- как формируются общие и индивидуальные настройки;
- как работает batch mode;
- как работает safe mode;
- как считается preflight estimate;
- как новый интерфейс подключается без переписывания `core`.
